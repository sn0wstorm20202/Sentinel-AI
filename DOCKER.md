# Docker Architecture - Sentinel AI

## 1. Multi-Stage Builds
Sentinel AI utilizes multi-stage builds to strictly isolate compilation/build dependencies from the final production images.

### Backend (`Dockerfile`)
- **Stage 1 (Builder):** Uses `python:3.10-slim`. Installs `build-essential` and compiles Python packages into `.whl` files.
- **Stage 2 (Runner):** Uses `python:3.10-slim`. Copies only the wheels from Stage 1, creates a non-root `sentinel` user, and boots via `gunicorn`.

### Frontend (`frontend/Dockerfile`)
- **Stage 1 (Builder):** Uses `node:20-alpine`. Installs dependencies using `pnpm` and executes `next build`.
- **Stage 2 (Runner):** Uses `node:20-alpine`. Copies ONLY the `.next/standalone` directory generated by Next.js. Executes as a non-root `nextjs` user.

## 2. Docker Compose
The `docker-compose.yml` bridges the architecture:
- **`backend`:** Exposes port 8000 internally. Checks `/live` for health.
- **`frontend`:** Exposes port 3000 internally. Depends on `backend` being strictly healthy before starting.
- **`nginx`:** Exposes port 8080 to the host. Maps internal Nginx routing to the respective containers.
